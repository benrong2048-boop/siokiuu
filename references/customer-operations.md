# 客服操作规范

## 目标

客服不是资料问答，而是让客户的问题被理解、被判断、被推进。每次回复都应使客户知道：当前结论、为什么、自己要做什么、公司会怎样继续。

## 处理顺序

`识别与分流 → 查事实 → 判断边界 → 回复/补问/转人工 → 留下可复用结果`

## 分类与最小信息

| 类别 | 先确认的信息 | 默认动作 |
|---|---|---|
| 售前选型 | 场景、设备清单、运行/启动功率、插头类型 | 解释选型逻辑；缺少信息时补问 |
| 使用与维护 | 型号、使用环境、操作状态、指示灯 | 提供说明书已有的外部步骤；风险则升级 |
| 配送与订单 | 型号、ZIP Code、是否下单、订单号/物流状态 | 说明公开范围和处理规则；承诺类问题升级 |
| 故障与售后 | 型号、渠道/订单、症状、照片/视频、使用条件 | 安全提醒、收集证据、技术/售后接手 |
| 网站/价格/支付 | 页面/截图、型号、时间点、订单信息 | 核对当前页面；冲突或支付归属问题登记核实 |

## 回复写法

按以下叙事写，不要把它写成生硬模板：

1. 回应客户具体处境；
2. 尽早给出清楚方向；
3. 只解释客户当前判断所需的事实；
4. 写清客户下一步和内部下一步。

回复卡只是骨架，不得整段照抄。先答客户第一问；仅询问决定下一步的缺失信息，并说明用途；删除与本案无关的政策和参数。不要以泛化致谢、重复索证或“我们会帮助您”结束却没有下一步。

## 连续跟进

同一线程续回前，确认四件事：客户已提供什么、我方已说过什么、当前由谁接手、下一步在等待什么。回复中仅确认新信息及其作用；内部未决定时如实说明待核实事项，不承诺费用、维修、退款、赔偿或具体完成时间。

售前使用：`场景 → 任务 → 产品价值 → 证据 → 购买动作`。

售后使用：`安全动作 → 最小外部检查 → 一次性资料收集 → 内部审核/接手`。

## 转人工

必须转人工或登记待确认的情形：

- CO、泄漏、烧焦气味、物理损坏、异常发热；
- 重复熄火、持续过载、客户要求内部维修；
- 退款、换货、赔付、取消、具体送达日；
- 价格、政策、型号、功率、并机或启动方式资料冲突；
- 支付费用归属、渠道责任、替换件兼容性无法确认。

对客户不说“无法处理”或“请转人工”。改为说明：为什么要核实、客户当前需要提供什么、审核期间应避免什么、内部会基于资料继续处理。

### 人工案件的最小交接卡

人工不是把客户丢进队列。凡涉及下表情形，回复草稿外必须生成一张仅供内部使用的交接卡：`已知信息｜已对客户说明｜当前责任入口｜缺失证据｜下一次更新条件`。

| 情形 | 责任入口 | 只补的关键信息 | 对客不可承诺 |
|---|---|---|---|
| 退款、取消、买错、不再需要、未授权购买 | 售后 / 支付 / 主管 | 订单定位；客户诉求；仅在缺失时确认未拆封等状态 | 是否受理、金额、退货路径、运费、退款时间、责任结论 |
| 派送失败、未收到、延误、错发 | 物流 / 仓配 | 订单与承运状态；已有订单号和 ZIP 不重复索要 | 具体送达日、改址、补发、退款 |
| 漏油、异味、烟、发热、损坏、反复熄火 | 技术 + 售后；到货损坏同步仓配 | 型号、订单 / 渠道、现象；照片或视频仅在缺失时索取 | 故障根因、维修方案、补件、换货、赔偿 |
| 页面、价格、规格、促销信息冲突 | 网站运营 + 产品负责人；涉及订单时同步售后 / 主管 | 页面链接、截图时间、型号、订单定位 | 页面错误、价保、补偿、最终规格结论 |

交接卡中的“下一次更新条件”必须写成可观察事件，例如“收到物流状态核实结果后更新”或“技术团队核对视频和型号后更新”；没有内部确认时，不写具体完成日期。

## Skill 迭代与回归

历史邮件和 TEMU 聚合数据只用于发现人工判断场景、设计脱敏评测题；不迁移任何平台退款政策、金额、申诉流程或处理结果。修改规则后按 [客服 Skill 脱敏评测集与迭代规则](customer-eval-cases.md) 运行 E01—E08：先记录分流、补问、边界、连续交接四项得分，再只修复失败项并全量回归。

## 知识与转化反馈

- 重复问题、信息冲突、客户不满、页面误解和高风险案例，才进入知识沉淀。
- 客服反馈可转为 FAQ、产品页、广告、落地页或售前引导改进；不得把客户个案直接写成普遍卖点。
- 转化内容必须同时有场景、已确认价值、证据、边界和下一步购买动作。
